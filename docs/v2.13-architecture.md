# VEGA v2.13 Architecture - Controlled Coding Workflows

Maintained release: `v2.13.0`

## Invariant and scope

VEGA v2.13 coordinates existing deterministic components; it is not an
unrestricted autonomous coding agent.

```text
Code and deterministic policy decide.
The model may analyze and explain.
The model never receives direct tool authority.
```

The controlled path remains contextual routing, deterministic interpretation
and planning, `PlanExecutor`, `ToolExecutor`, `PermissionEvaluator`, and a
registered handler. The workflow engine reuses Patch Tools, Test Tools,
permission policy, checkpoints, execution traces, diagnostics, and the v2.12
generated-state lock. It introduces neither a second tool registry nor a model
callable tool surface.

## Workflow types

* `bug-fix`, `feature`, and `refactor` perform bounded read-only investigation,
  wait for a managed Patch Tools artifact, and gate patch and tests separately.
* `test` resolves one configured allowlisted group and runs it exactly once only
  after confirmation. `all` remains a distinct group and is never inferred from
  a focused request.
* `review` reads a bounded unstaged or explicitly staged Git diff and emits only
  deterministic structured findings. It never applies a patch.
* `bugfix` is retained as an input alias for older callers and normalizes to
  `bug-fix` in schema-v2 state.

## Lifecycle and legal transitions

| Current stage | Explicit action or event | Next stage | Side effect |
| --- | --- | --- | --- |
| none | `start` coding workflow | `investigating` | read-only inspection |
| `investigating` | bounded investigation completes | `waiting_patch` | none |
| `waiting_patch` | attach managed pending patch | `awaiting_patch_confirmation` | none |
| `awaiting_patch_confirmation` | exact patch approval after binding and drift checks | `awaiting_test_confirmation` | apply once |
| `awaiting_test_confirmation` | exact test approval after binding and drift checks | `tests_running` | run configured group once |
| `tests_running` | tests pass | `completed` | archive evidence |
| `tests_running` | focused tests fail below iteration cap | `waiting_patch` | preserve patch and outcome |
| `tests_running` | drift, unsafe result, standalone failure, or cap | `failed` | archive evidence |
| legal waiting stage | `cancel` | `cancelled` | no automatic rollback |
| applied state with unchanged workspace | exact `rollback` | `rolled_back` | Patch Tools rollback once |
| interrupted pre-action state | `resume` | same safe waiting stage | no replay |
| interrupted apply whose managed patch is already applied | `resume` | `awaiting_test_confirmation` | reconcile only |
| interrupted `tests_running` | `resume` | `failed` | never rerun tests |

Only one active workflow is allowed. Terminal states are archived under the
existing `data/workflows/history` directory. Invalid transitions, extra command
arguments, fuzzy commands, and replay are rejected without action execution.

## Confirmation binding

`ConfirmationBinding` is frozen, slotted, manually serialized, and integrity
hashed. It binds the exact workflow ID, stage, action, patch identity, workspace
revision, test group, command ID, state revision, and creation time. The binding
is persisted immediately before the corresponding user-visible approval point.

Patch approval consumes only `patch_application`; the resulting state creates a
new `test_execution` binding. A confirmation cannot cross workflows, patches,
commands, stages, revisions, or later actions. Workspace or patch-identity drift
clears the pending binding and returns a fixed safe error. Ordinary language,
silence, model output, and a previous patch proposal never constitute approval.

Rollback uses the exact `/workflow rollback <workflow_id>` command as its
explicit single-use request. It is permitted only when Patch Tools proves the
managed patch is applied and the complete workspace revision is unchanged.

## Persistence, locking, and recovery

Schema-v2 state uses frozen slotted models and exact-field `to_dict`/`from_dict`
methods. The state contains bounded hashes, IDs, relative paths, counts,
booleans, fixed codes, and structured outcomes only. It never contains raw task
text, prompts, diff or file content, test output, stdout/stderr, tool arguments
or results, exception text, tracebacks, absolute paths, environment values,
confirmation tokens, secrets, callables, or arbitrary representations.

Writes use unique same-directory temporary files, flush, file `fsync`, atomic
replacement, and directory `fsync` on POSIX. Mutations share the v2.12
`GeneratedStateLock` with fixed `.workflow-state.lock`, in-process ordering, OS
locks, and bounded timeout. Reads reject symlinks, unexpected filenames,
multiple active files, oversized data, malformed JSON, unknown fields, and
incompatible schemas.

Checkpoints accept v1 and v2 payloads. Recovery restores only the latest safe
checkpoint and never executes a patch, test, model, tool, or shell operation.
`resume` performs the separate fail-closed reconciliation. A v1 `waiting_patch`
state migrates to sanitized schema v2; ambiguous executing/confirmation states
become `failed/state_incompatible` and cannot advance. Raw legacy task/context
data is removed when the migrated active state is resumed or next mutated.

## Patch and workspace safety

Managed patch metadata is reloaded at approval and matched to its bound SHA-256
identity. Paths must be normalized project-relative values under both Windows
and POSIX semantics. Absolute, drive-qualified, UNC, parent traversal, alternate
stream, root escape, symlink, `.git`, generated state, cache, virtual
environment, `node_modules`, and sensitive targets are rejected.

The workspace revision covers Git status plus the current commit when a Git
worktree is available. A bounded content hash is used for isolated fixtures.
Any change between proposal, patch approval, test approval, or post-test
evaluation stops the operation. Rollback refuses rather than overwriting
unrelated edits.

## Bounded iterations and test evidence

The compiled maximum is three patch/test iterations. Every patch and every test
run has a new confirmation. A failed focused run retains all prior investigation
evidence, applied patch metadata, a structured outcome, iteration count,
rollback flag, and safe next actions. Results retain only group/command IDs,
pass flag, return code, timeout flag, bounded duration, and a fixed outcome code.

## Diagnostics and traces

Doctor diagnostics inspect workflow state read-only and report only counts,
workflow/type/stage identifiers, next actions, confirmation state, iteration,
drift, rollback, and fixed codes. They do not instantiate a state store or create
workflow files.

Opt-in execution traces may contain a frozen `WorkflowTraceDecision` with the
same payload-free decision metadata. Records without this optional field remain
valid, preserving public v2.10/v2.11/v2.12 trace compatibility. Trace failures
are best effort and cannot change workflow results.

## Threat model

The design resists malicious task text, misleading model output, stale/replayed
approval, patch substitution, workspace drift, path confusion, symlink escape,
corrupt or oversized state, torn writes, competing VEGA processes, interrupted
actions, unsafe rollback, and diagnostic leakage. OS compromise, a malicious
process with direct write access, non-cooperating lock users, and forensic
recovery are outside scope; locks remain advisory.

## Known limitations

* Investigation is bounded and deterministic, not a guarantee of root-cause
  correctness.
* A managed patch must already be proposed through Patch Tools; the workflow
  does not accept arbitrary generated file content.
* Review findings use a small deterministic rule set and are not a substitute
  for expert review.
* Rollback is intentionally conservative and unavailable after any unrelated
  workspace change.
* Windows symlink tests skip explicitly when the account lacks symlink rights.
* No background work, arbitrary shell, internet autonomy, automatic Git write,
  telemetry, permission change, plugin installation, or in-product publication
  was added.

## Validation

The release gate covers lifecycle, confirmation binding, drift and replay,
focused/full test scope, three-iteration cap, cancellation, resume, interrupted
actions, corruption/oversize/migration, concurrent reads and mutations, lock
timeout, atomic-write failure, rollback and refusal, exact CLI routing, portable
paths, symlinks, protected directories, payload sentinels, trace/diagnostic
isolation, compatibility, compileall, release identity, production policy,
smoke, Release Manager, whitespace, generated-state hygiene, and the complete
Windows/Linux Python 3.12-3.14 CI matrix.

The local Windows/Python 3.12 gate recorded `233 passed, 4 skipped, 21 subtests`
focused and `1017 passed, 7 skipped, 136 subtests` full. All seven skips are
explicit unavailable-symlink-privilege conditions; five predate v2.13 and two
cover the new workflow/file path layer.
