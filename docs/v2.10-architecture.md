### Execution trace implementation contract

`core/execution_trace.py` owns the frozen `ExecutionTrace` and `TraceStep`
models, the request-local `TraceRecorder`, allowlist serialization, opt-in
store, bounded rotation, loading, and safe summaries. `core/plan_executor.py`
emits only `StepExecutionObservation` values: step id, tool name, permission,
risk, status, and safe error code. It never passes arguments, results,
exceptions, handlers, grants, or confirmation tokens to tracing.

Persistence is disabled unless `VEGA_EXECUTION_TRACE` is explicitly set to
`1`, `true`, `yes`, or `on`. Enabled traces append as UTF-8 JSONL to
`logs/diagnostics/execution-traces.jsonl`; the active file rotates at 5 MiB to
one `.1` backup. The store uses a process-local lock and does not claim
interprocess locking. Serialization, callback, append, rotation, and corrupt
record failures are diagnostic-only and never change the request result.

## 1. Scope

v2.10 is a stabilization release. It connects and validates the routing,
domain, capability, plugin, permission, model-selection, context-budget, and
release layers introduced through v2.7–v2.9. It does not add an autonomous
execution loop or a new user-facing feature family.

The completed release is planned to deliver:

1. one validated runtime snapshot used by routing and execution;
2. cross-layer policy consistency checks at startup, plugin bootstrap,
   `/doctor`, tests, and release checks;
3. explicit runtime-state ownership and Git hygiene;
4. a bounded, redacted execution trace of machine decisions;
5. deterministic model-unavailable and synthesis-failure behavior;
6. integration and regression coverage for the complete request path; and
7. synchronized release identity and documentation.

This document is based on the repository as it exists. Two paths named in the
planning brief do not exist: `core/result_synthesizer.py` and
`tools/schemas.py`. Their current responsibilities are implemented by
`core/contextual_synthesis.py`, `core/contextual_response.py`, and the immutable
models colocated with the relevant core, plugin, domain, and permission
modules. v2.10 must not introduce placeholder files merely to match those
names.

### 2.2 Actual request path

The current interactive path is split across two routing systems:

```text
scripts/vega.py
  -> core.agent_runtime.main
  -> AgentOrchestrator
       -> IntentRouter for empty/command/confirmation/chat/workflow routing
  -> for eligible chat requests: try_execute_contextual_request
       -> analyze_intent
       -> load model policy and select_model
       -> route_contextual_request
            -> interpret_task
            -> build_tool_catalog(TOOL_REGISTRY, tool_capabilities.json)
            -> plan_tools
       -> execute_plan
            -> ToolExecutor
                 -> PermissionEvaluator
                 -> registered handler
       -> deterministic formatting
       -> optional bounded synthesis for document_analysis/code_review
  -> otherwise: ordinary Ollama chat
```

The language model is not given `ToolExecutor`, the registry, confirmation
tokens, or callable handlers. This boundary is correct and remains invariant.

### 2.3 Cross-layer gaps

* `DomainRegistry` is not consulted by `route_contextual_request`; built-in
  domain intent/capability/tool metadata can therefore drift from
  `_INTENT_ROUTES` and `tool_capabilities.json` without preventing startup.
* `bootstrap_plugins` validates plugin activation and can build a combined
  mapping, but `build_production_tool_executor` always uses the compatibility
  `TOOL_REGISTRY`. Production startup does not bootstrap plugins.
* `tools.registry.BUILTIN_TOOL_REGISTRY` is an immutable view, while
  `TOOL_REGISTRY` is a mutable compatibility copy. Construction is safe only
  if one validated snapshot is passed consistently to catalog, permissions,
  and executor.
* Permission policy alignment validates registry names exactly, but there is no
  single check spanning intent, domain, capability, routing metadata, plugin
  activation, permissions, and model policy.
* Most policy loaders are fail closed, but they do not share schema-version
  validation or a common diagnostic report. `tool_routing_policy.json` and
  `model_routing_policy.json` currently have no `schema_version` field.
* `ToolExecutor` permits an evaluator-free custom registry for isolated legacy
  tests. Production factories are safe, but v2.10 tests must ensure no
  production/plugin path constructs that form.
* An unknown contextual intent currently returns `NOT_HANDLED` and may fall
  through to chat. A distinction is needed between ordinary chat and a request
  that asks for execution but cannot be classified.
* Model selection marks an explicit model override as available without
  checking the installed-model set. Other candidates are checked.
* Successful read-only evidence already survives synthesis failure because the
  deterministic response is retained. This correct behavior needs a
  cross-layer regression test and trace status.

### 2.4 Resolved runtime-state and release gaps

* `data/model_profile.json` and `data/project_state/tasks.json` remain local but
  are ignored and untracked; index removal never deletes their local contents.
* Mutable indexes, memory, patches, task archives, workflows, checkpoints, and
  logs retain explicit ignore ownership.
* Profile writes use same-directory atomic replacement and deterministic
  temporary-file cleanup.
* `scripts/version.py`, runtime/CLI identity, README, CHANGELOG, roadmap,
  architecture, commands, security documentation, and release notes identify
  `v2.10.0`; v2.8 and v2.9 have coherent historical CHANGELOG sections.
* Release-identity tests reject known mojibake markers in the maintained release
  documents.

## 3. Problems being solved

1. A route can be locally valid while contradicting domain or model metadata.
2. Plugin tools have a supported safe executor factory but are not part of a
   single production bootstrap.
3. Invalid or missing configuration is reported by individual loaders rather
   than one actionable, severity-ranked report.
4. Runtime decisions are difficult to diagnose without reading logs and code.
5. Mutable user/runtime files can pollute Git or be mistaken for fixtures.
6. Model availability and fallback outcomes are not uniform at every priority
   level.
7. Release claims are not mechanically synchronized.

## 4. Non-goals

v2.10 does not add GUI, monitoring, telemetry export, a marketplace, plugin
installation, hot reload, a new planner, a second tool registry, a second
permission system, arbitrary shell generation, autonomous retries, model-driven
tool invocation, or automatic publishing. It does not sandbox trusted Python
plugins. It does not redesign workflows, RAG, memory, or the command system.

Historical encoding cleanup and a general configuration framework are deferred.
Runtime state is not relocated outside the project in this release because that
would create a broad migration and portability change.

## 5. Architectural invariants

1. **Code decides; the model explains.** The model may analyze bounded text or
   synthesize an answer. It never receives `ToolExecutor`, a callable registry,
   `PermissionEvaluator`, confirmation tokens, secrets, or authority to advance
   a plan.
2. **One controlled tool path.** Every built-in or active plugin call flows
   through registry lookup, capability/route validation, permission evaluation,
   confirmation rules, Python signature binding, handler invocation, and a
   structured result.
3. **One runtime snapshot.** The catalog, executor, diagnostics, and trace use
   the same immutable combined tool mapping and loaded policies.
4. **Fail closed for execution ambiguity.** An execution-like request with an
   unknown intent, missing domain/capability/tool/rule, invalid policy, unsafe
   path, unavailable model when synthesis is required, or unconfirmed dangerous
   action cannot execute.
5. **Plain chat remains plain chat.** A request classified by `IntentRouter` as
   ordinary chat may reach the model. “Unknown intent” is fatal only after the
   request has been classified as an execution candidate; this avoids treating
   general conversation as a tool failure.
6. **No partial execution after failed preflight.** All plan steps are checked
   against the snapshot and automatic-permission set before step one.
7. **Evidence is durable within the result.** Synthesis cannot mutate or erase
   successful `StepExecutionResult` values.
8. **Runtime state is local.** Normal execution and profile selection do not
   modify tracked files.
9. **Diagnostics expose decisions, not reasoning.** Trace data contains enums,
   identifiers, counts, bounded reasons, and safe error codes—not prompts,
   chain-of-thought, secrets, tokens, or unrestricted payloads.
10. **Release actions remain manual.** Policy continues to forbid automatic
    commit, tag, push, and GitHub release creation.

### 6.2 New component: `core/policy_consistency.py`

This file is required because consistency spans otherwise independent modules
and cannot belong to any one loader without reversing dependency direction.

Public API:

```python
class ConsistencySeverity(str, Enum):
    FATAL = "fatal"
    DEGRADED = "degraded"
    WARNING = "warning"

@dataclass(frozen=True, slots=True)
class ConsistencyIssue:
    code: str
    severity: ConsistencySeverity
    component: str
    message: str
    subject: str = ""

@dataclass(frozen=True, slots=True)
class PolicyConsistencyReport:
    issues: tuple[ConsistencyIssue, ...]
    checked_intents: tuple[str, ...]
    checked_domains: tuple[str, ...]
    checked_tools: tuple[str, ...]
    schema_versions: Mapping[str, int]

    @property
    def fatal(self) -> bool: ...
    @property
    def degraded(self) -> bool: ...
    def to_safe_dict(self) -> dict[str, object]: ...

def validate_policy_consistency(
    *,
    intents: Iterable[str],
    intent_capabilities: Mapping[str, tuple[str, ...]],
    domains: DomainRegistry,
    tool_mapping: Mapping[str, Callable[..., object]],
    tool_capabilities: Mapping[str, object],
    permission_policy: PermissionPolicy,
    model_policy: ModelRoutingPolicy,
    routing_policy: ToolRoutingPolicy,
    plugin_activations: Iterable[PluginToolActivation] = (),
    schema_versions: Mapping[str, int],
) -> PolicyConsistencyReport: ...
```

Inputs are already parsed immutable policies and the combined tool snapshot;
the validator performs no imports, tool execution, model call, or state write.
The production bootstrap loads files and then calls this pure function.

Dependencies allowed: `core.intent_analyzer`, `core.model_selection`,
`core.contextual_router`, `core.tool_planner`, `domains`, `plugins.models`, and
`permissions.models`. It must not import `agent_runtime`, `ToolExecutor`, tool
handlers, Ollama, workflows, or CLI modules.

Checks and severity:

| Condition | Severity | Code |
|---|---|---|
| actionable intent has no enabled domain | fatal | `intent_domain_missing` |
| domain intent has no planner route | fatal | `intent_route_missing` |
| domain has no capability or required capability is not owned | fatal | `domain_capability_missing` |
| required capability has no configured active tool | fatal | `capability_tool_missing` |
| configured tool is absent from combined mapping | fatal | `tool_not_registered` |
| active tool has no exact permission rule | fatal | `tool_permission_missing` |
| permission rule names an absent tool | fatal | `permission_tool_missing` |
| plugin tool lacks routing metadata | fatal while plugin enabled; warning while disabled | `plugin_routing_missing` |
| built-in/plugin tool collision | fatal | `tool_name_collision` |
| model intent profile is absent or unknown | fatal | `model_profile_missing` |
| fallback profile/order references unknown profile | fatal | `model_fallback_missing` |
| context budget is missing for a profile | fatal | `context_budget_missing` |
| route points to a disabled domain | fatal | `domain_disabled` |
| supported policy schema differs | fatal | `policy_schema_mismatch` |
| registered tool intentionally has no automatic route | warning | `tool_not_auto_routed` |
| disabled plugin has stale manifest-only metadata | warning | `disabled_plugin_metadata` |

`degraded` is reserved for an optional subsystem that can be excluded without
changing a previously valid built-in route—for example, an allowed plugin that
fails to import when policy explicitly permits partial plugin availability.
The v2.10 default plugin policy is atomic and fail closed, so current plugin
bootstrap failures are fatal. Severity must never downgrade a permission,
schema, built-in route, or model-policy error.

### 6.3 New component: `core/execution_trace.py`

This file is required because trace lifecycle crosses routing, execution,
selection, budgeting, and synthesis, while none of those components should own
persistence or redact other layers’ data.

Public API:

```python
class TraceStatus(str, Enum):
    STARTED = "started"
    COMPLETED = "completed"
    BLOCKED = "blocked"
    FAILED = "failed"

@dataclass(frozen=True, slots=True)
class TraceStep:
    step_id: int
    tool_name: str
    permission: str
    risk: str
    status: str
    error_code: str = ""

@dataclass(frozen=True, slots=True)
class ExecutionTrace:
    trace_id: str
    request_type: str
    intent: str
    domain: str
    required_capabilities: tuple[str, ...]
    selected_tools: tuple[str, ...]
    permission_outcomes: tuple[str, ...]
    confirmation_required: bool
    model_profile: str
    model: str
    model_reason_code: str
    fallback_used: bool
    context_budget: Mapping[str, int | bool]
    steps: tuple[TraceStep, ...]
    status: TraceStatus
    error_codes: tuple[str, ...]

    def to_safe_dict(self) -> dict[str, object]: ...

class TraceRecorder:
    def record_route(...): ...
    def record_model(...): ...
    def record_step(...): ...
    def record_synthesis(...): ...
    def finalize(...) -> ExecutionTrace: ...

def serialize_trace(trace: ExecutionTrace, *, max_chars: int = 262_144) -> str: ...
def append_trace(project_root: Path, trace: ExecutionTrace) -> Path: ...
def load_latest_trace(project_root: Path) -> ExecutionTrace | None: ...
```

`TraceRecorder` is request-local mutable assembly state; `finalize` returns the
immutable trace and may be called once. The trace is created before contextual
routing, enriched only from structured code decisions, and finalized on every
handled success, block, or failure. Plain chat may record only request type and
model status when tracing is explicitly enabled.

Allowed dependencies are immutable result/model modules and the standard
library. It must not import tool handlers, registry construction, plugin loader,
`agent_runtime`, prompts, memory, or RAG.

## 7. Request lifecycle

The v2.10 lifecycle is:

```text
raw request
  -> AgentOrchestrator: command / confirmation / workflow / chat
  -> execution-candidate classification
  -> intent analysis
  -> domain resolution (exactly one enabled owning domain)
  -> required capability lookup
  -> validated runtime snapshot lookup
  -> deterministic tool planning
  -> immutable ExecutionPlan validation
  -> whole-plan permission/risk/confirmation preflight
  -> ToolExecutor for each step
  -> immutable evidence-bearing step results
  -> model selection
  -> evidence context budget
  -> optional synthesis
  -> deterministic fallback or synthesized user response
  -> finalized redacted ExecutionTrace
```

Startup creates one `RuntimeBootstrapResult` in the existing
`core/tool_executor_factory.py` (a new frozen dataclass in that file, not a new
module). It contains the built-in domain registry, plugin bootstrap result,
combined immutable tool mapping, permission evaluator, session grants, routing
and model policies, consistency report, and executor. `agent_runtime.main`
passes its mapping to catalog/routing and its executor to plan execution.

The runtime does not expose this object to the model. Fatal consistency issues
prevent the interactive ready state. `/doctor` remains available through a
minimal diagnostic startup path that loads no plugin handlers after a fatal
pre-bootstrap policy error.

## 8. Policy consistency model

### 8.1 Authoritative relationships

* `IntentType` and `_INTENT_ROUTES` define actionable built-in intent routes.
* enabled `DomainDefinition` values own each routable intent and declare the
  broader capabilities and tool-name boundary permitted for that domain.
* `tool_capabilities.json` maps a registered tool to routing permission and
  fine-grained route capabilities.
* the combined tool snapshot proves the handler exists and has unique
  provenance.
* `permission_policy.json` authorizes or denies invocation risk and system
  capabilities.
* `model_routing_policy.json` maps every contextual intent (including unknown)
  to a known profile and budget.

These vocabularies are not identical: routing permissions (`READ`, `DRAFT`,
etc.), route capabilities (`git.diff`), domain capabilities (`git.read`), and
permission capabilities (`git.read`, `process.execute`) serve different
purposes. v2.10 must document and validate an explicit mapping instead of
assuming string equality between all layers. The least invasive implementation
adds `route_capabilities` to `DomainDefinition` or normalizes existing domain
capabilities to the exact routing vocabulary. The preferred change is to make
domain `capabilities` use the same fine-grained values as the planner/catalog
and keep security capability classification solely in permission rules.

### 8.2 Invocation points

* **Startup:** full validation after plugin manifests are collected and before
  publishing executor/ready state.
* **Before plugin bootstrap publication:** validate prospective combined
  metadata; never publish a partial set.
* **`/doctor policies`:** load and report without executing a tool or model.
* **Release check:** fail on fatal/degraded findings and optionally display
  warnings.
* **Tests:** validate the real production snapshot plus isolated invalid
  fixtures.

### 8.3 Schema versions

All six cross-layer JSON policies must contain integer `schema_version: 1`:
tool routing, tool capabilities, plugin, permission, model routing, and release.
Loaders reject missing, boolean, unsupported, or extra schema values. v2.10
does not create a generic schema engine; each loader remains authoritative for
its fields and the consistency report verifies the supported versions agree.

does not silently rewrite corrupt state; profile state may safely fall back in
memory and is rewritten only after the user next selects auto/manual mode.

## 10. Execution Trace

### 10.1 Data and lifecycle

The immutable trace contains only:

* normalized request type, not full request text;
* selected intent and domain;
* required capability names and selected tool names;
* permission outcome labels, risk labels, and confirmation requirement;
* selected profile/model, stable reason code, and fallback flag;
* context budget counts (`original_chars`, `selected_chars`, `max_chars`,
  `truncated`);
* step IDs, tool names, status, and safe error codes;
* final status.

It never contains request/evidence/tool payloads, system prompt, model messages,
hidden reasoning, confirmation token, environment variables, URLs with query
strings, unrestricted paths, or exception representations that may embed data.

Lifecycle transitions are monotonic:

```text
STARTED -> COMPLETED
STARTED -> BLOCKED
STARTED -> FAILED
```

After finalization no field changes. Step count is capped by the already
validated plan `max_steps` (production maximum 8). String identifiers are
limited to 128 characters, reason/error codes to a fixed allowlist, issue
messages to 500 characters, and serialized trace records to 256 KiB.

### 10.2 Redaction and serialization

`to_safe_dict` constructs an allowlisted dictionary; it must not recursively
serialize arbitrary objects. `serialize_trace` uses UTF-8 JSON with sorted keys
and rejects an oversized result rather than truncating JSON. Unknown exception
text maps to a stable code such as `internal_error`; detailed exceptions remain
only in existing local logs subject to their own policy.

Persistence is disabled by default. `VEGA_EXECUTION_TRACE=1` enables append-only
storage at `logs/diagnostics/execution-traces.jsonl`. The writer uses a process
lock, a maximum 5 MiB active file, and one `.1` rotation; write failure never
changes the request result but is surfaced as a warning. This is bounded local
diagnostics, not a monitoring engine.

CLI additions reuse `handle_doctor_command`:

* `/doctor policies` — current consistency report;
* `/doctor trace latest` — latest safely parsed persisted trace or “disabled / no trace”; and
* `/doctor` — summary counts and trace-enabled state.

## 11. Failure model

| Failure | Outcome | Evidence/trace rule |
|---|---|---|
| invalid/missing policy or schema mismatch | fatal startup / handled failure | no execution; safe policy code |
| execution candidate with unknown intent | blocked | no chat fallback and no tool |
| plain conversational chat | model chat path | not an execution failure |
| missing/disabled domain | blocked | domain code only |
| missing capability/tool/rule | blocked | whole-plan preflight, zero steps run |
| plugin provenance/import/collision error | fatal under atomic default | no combined snapshot published |
| permission deny/missing confirmation | blocked | no handler invocation |
| invalid arguments/path | failed | safe code; no raw argument in trace |
| step or tool-reported failure | failed | previous step results retained |
| step limit exceeded | blocked during plan validation | zero steps run |
| Ollama unavailable | deterministic model-unavailable decision | tools that do not require synthesis may still return deterministic evidence |
| selected/fallback model absent | deterministic unavailable result | no model call |
| evidence over budget | bounded synthesis input | original step evidence remains unchanged |
| synthesis failure | completed tool result with deterministic response | trace records `synthesis_failed`; status remains completed |
| trace persistence failure | request result unchanged, diagnostic warning | trace may remain in memory |

Fatal errors prevent publication of a runtime snapshot. Degraded errors exclude
only a policy-declared optional plugin and cannot affect built-in routes.
Warnings never grant capability or permission.

## 12. Security boundaries

* `scripts/vega.py` remains an entrypoint only.
* `AgentOrchestrator`, intent analysis, planning, selection, budgeting, and
  synthesis never import or receive callable tool handlers.
* Only composition code in `tool_executor_factory.py` and plugin runtime code
  may assemble mappings and evaluators.
* Active plugin handlers enter the same combined mapping passed to the existing
  `ToolExecutor`; there is no plugin-side invocation API.
* Plugin import is trusted-code execution, not sandboxing. Exact module,
  prefix, trusted-root, source-loader, and origin checks remain mandatory.
* Production construction always supplies `PermissionEvaluator`. The
  evaluator-free `ToolExecutor(custom_registry)` compatibility form remains
  test-only and must be documented as unsupported for production/plugin use.
* Confirmation tokens and session grants never enter plans, traces, model
  messages, or plugin manifests.
* Trace and synthesis use explicit allowlists and size bounds.

## 13. Model-selection behavior

Priority is deterministic:

1. explicit model/profile override supplied by trusted runtime code;
2. manually selected stored profile;
3. automatic intent/complexity profile;
4. first installed model in validated fallback order; and
5. unavailable decision with empty model and stable reason code.

Every candidate, including an explicit override, is checked against the
installed-model snapshot. `explicit_model` is not user free text passed from a
model; it is a trusted caller option. A literal installed model outside
`MODEL_PROFILES` may be accepted only when policy explicitly permits external
model names; v2.10 default is false.

Behavior table:

| Condition | Decision |
|---|---|
| Ollama unavailable | installed set is empty; deterministic unavailable, no subprocess retry loop |
| requested model absent | evaluate validated fallback order |
| all fallbacks absent | `available=False`, empty model, `models_unavailable` |
| corrupt/missing model policy | fatal policy error; no implicit defaults in production |
| unknown stored profile | safe local-state default `code`, plus trace warning; policy still validated |
| plugin intent absent from model policy | plugin tool remains inactive/fatal consistency while plugin is enabled |
| evidence exceeds budget | head/tail bounded copy goes to synthesis; original evidence retained |
| synthesis fails | deterministic formatter returns successful read-only tool evidence |

Model selection should occur after route/domain validation and before any step
only when the route requires model synthesis. For current deterministic
read-only routes, tool execution is not made dependent on Ollama availability.
This prevents a missing model from discarding useful local evidence.

## 14. Plugin compatibility

The v2.8 manifest, provenance, collision, and two-phase publication contracts
remain intact. v2.10 changes composition, not the public plugin schema.

Production bootstrap order:

1. load built-in domains and immutable built-in tools;
2. strictly load all policies;
3. load allowed plugin manifests under existing provenance checks;
4. compute activation from domain, route capability, and permission metadata;
5. create the prospective combined mapping;
6. run cross-layer consistency on the entire prospective snapshot;
7. publish the snapshot only if no fatal issue exists; and
8. construct one `ToolExecutor` with one evaluator/session-grant store.

Denied, mismatched, or disabled plugin tools remain outside the mapping.
Partial bootstrap is disabled by default. No plugin can add a new intent unless
that intent is recognized by the analyzer/planner, owned by an enabled domain,
and covered by model routing and context budget policy. This intentionally
makes new intent families a coordinated code/policy release rather than a
manifest-only extension in v2.10.

## 18. Acceptance criteria

v2.10 is accepted only when:

* normal startup and model switching leave `git status` unchanged;
* mutable state, especially model profile and task state, is ignored/untracked
  but preserved locally;
* all policy files load strictly and the production consistency report has no
  fatal or degraded findings;
* every supported intent resolves to an enabled domain, required capabilities,
  registered tools, permission rules, model profile, and context budget;
* every built-in/plugin invocation uses the same permission-enforced executor;
* configuration, provenance, schema, path, and confirmation ambiguity fails
  closed before handler invocation;
* successful read-only evidence remains available after synthesis failure;
* execution traces are bounded, redacted, immutable after finalization, and
  disabled for persistence by default;
* all existing and new tests pass in a writable temporary environment;
* `python -m compileall` passes;
* README, CHANGELOG, roadmap, architecture, release notes,
  `scripts/version.py`, and release policy agree on v2.10.0; and
* no commit, tag, push, PR, or release is performed automatically.

## 19. Risks and trade-offs

* **Configuration coupling:** strict consistency means a new route often
  requires coordinated policy changes. This is intentional for a local safety
  agent; fixtures and stable error codes reduce maintenance cost.
* **Startup availability:** fail-closed validation can prevent normal startup
  for metadata drift. `/doctor` must remain usable through a non-executing
  diagnostic path.
* **Plugin trust:** shared routing and permissions do not sandbox import-time or
  direct Python side effects. Documentation must keep this limitation explicit.
* **Trace usefulness vs privacy:** payload-free traces cannot explain content
  quality, but they safely explain machine routing and policy decisions. Raw
  evidence remains in request-local results, not trace storage.
* **State compatibility:** atomic profile writes improve safety but Windows
  replacement and antivirus interference need explicit tests and controlled
  error handling.
* **Historical release debt:** synchronizing directly to v2.10 requires clear
  CHANGELOG entries for v2.8 and v2.9 without rewriting their Git history.
* **Capability vocabulary migration:** normalizing domain capabilities can
  break plugin manifests that copied broad v2.8 names. v2.10 should either
  support a documented one-release alias map or fail with precise migration
  diagnostics; silent coercion is unsafe.
