# VEGA v2.12.0 Release Notes

VEGA v2.12.0, **Local State Integrity & Recovery**, makes local execution traces
and diagnostic reports safe for cooperating concurrent VEGA processes and
recoverable after interrupted writes.

## Highlights

* Bounded dependency-free Windows and POSIX interprocess locks now protect trace
  append/rotation and atomic diagnostic-report export/retention.
* `/doctor state status` exposes a read-only, payload-free integrity summary.
* `/doctor state repair` explicitly recovers incomplete JSONL tails, quarantines
  complete corruption, removes only recognized stale temp files, and enforces
  bounded quarantine retention.
* Runtime diagnostics include a frozen, slotted, manually serialized local-state
  section with only relative paths, counts, booleans, and fixed safe codes.
* Diagnostics policy schema version 2 adds bounded lock, stale-temp, state-scan,
  and quarantine-retention controls.

## Security guarantees

Lock names are fixed inside validated generated-state directories. Configured
paths reject absolute and drive paths, traversal, blocked directories, and
symlink escape. Repair accepts no path, does not recurse, never runs at startup,
and preserves unrelated files. Raw payloads, prompts, exceptions, tracebacks,
absolute user paths, environment values, credentials, and tokens never enter
doctor output or serialized diagnostics.

All acquisition and repair work has hard limits and fixed failure codes. Atomic
writes use exclusive same-directory temp files, flush, `fsync`, and replacement.
Complete corruption is quarantined before valid trace records replace the source.

## Compatibility and migration

No trace API or stored trace record migration is required. Valid v2.10 and v2.11
JSONL records remain readable, and tracing remains disabled unless
`VEGA_EXECUTION_TRACE` is enabled. Existing report and trace paths remain the
defaults. Projects must use the supplied diagnostics policy schema version 2.

The execution architecture is unchanged:

```text
contextual runtime -> plan execution -> PlanExecutor -> ToolExecutor
```

Diagnostics remain observer-only and add no tool, model, network, telemetry,
background, or publishing authority.

## Recovery and rollback

Run `/doctor state status` first. Run `/doctor state repair` only when an explicit
local repair is desired. Repair is idempotent: clean state produces no changes;
content-addressed quarantines are not duplicated; and only recognized stale temp
files are removed. Quarantined originals remain under the generated diagnostics
directory subject to bounded retention.

For an application rollback, restore the v2.11 diagnostics policy schema. Valid
traces remain readable. Stop all VEGA processes before manually removing lock or
quarantine files.

## Known limitations and non-goals

* Locks are advisory and do not coordinate unrelated programs that ignore them.
* Recovery is bounded rather than forensic and will fail closed at scan limits.
* There is no startup repair, background monitor, remote telemetry, upload,
  autonomous execution, new tool path, or internal automatic publication.
* Filesystem integrity against privileged local attackers and hardware failure is
  outside the scope of this release.

## Validation

The release was validated with focused concurrency/recovery tests, the complete
test suite, compileall for source and tests, release identity and production
policy checks, smoke tests, built-in Release Manager checks, Git whitespace
checks, generated-state tracking checks, and GitHub CI. Detailed final counts and
the merge/tag/release pointers are recorded in the GitHub Release.

Local release-gate results on Windows were `106 passed, 2 skipped` for the
focused v2.12/release set and `1126` full-suite test results with zero failures,
zero errors, and five skips. Compileall, identity, production policy consistency,
smoke, Release Manager, Git whitespace, and generated-state tracking checks
passed. The five skips are conditional symlink tests on a Windows account without
symlink privileges; four predate v2.12 and the new lock/path symlink regression
uses the same platform condition. The two policy warnings are the intentional
nonautomatic `bug_fix` and `test_run` routes.
