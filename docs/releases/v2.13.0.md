# VEGA v2.13.0 Release Notes

VEGA v2.13.0 ships **Controlled Coding Workflows**: one deterministic,
restart-safe path over the existing workflow, Patch Tools, Test Tools,
permission, checkpoint, trace, diagnostics, and v2.12 state-integrity layers.

## Highlights

* `bug-fix`, `feature`, and `refactor` workflows perform bounded read-only
  investigation, attach a real managed patch, and require distinct approvals
  for application and tests.
* `test` resolves an allowlisted group and executes it once. Focused requests
  never expand silently to `all`.
* `review` inspects bounded staged or unstaged Git evidence and remains read-only.
* Stable IDs, explicit legal stages, next actions, status/detail/history,
  cancellation, resume, checkpoints, recovery, and conservative rollback are
  exposed through exact case-sensitive `/workflow` commands.
* Failed focused tests preserve applied-patch evidence and rollback options for
  at most three separately confirmed repair iterations.

## Security boundaries

Confirmations are single-use bindings over workflow, stage, action, patch
identity, workspace revision, test group, command ID, and state revision. Patch
approval cannot authorize tests. Drift, patch substitution, replay, wrong
workflow, missing policy/configuration, unsafe results, and unrelated edits fail
closed.

Workflow state, diagnostics, and traces use frozen slotted models and manual
allowlisted serialization. They exclude raw tasks, prompts, conversation, diff
or file content, test output, stdout/stderr, tool arguments/results, exception
text, tracebacks, absolute paths, environment data, tokens, credentials,
confirmation material, callables, and arbitrary representations.

Paths are checked under Windows and POSIX rules and protect `.git`, VEGA state,
caches, virtual environments, and `node_modules`. Writes are atomic and locked;
unknown, corrupt, oversized, symlinked, or incompatible workflow state cannot
advance or execute.

The model still has no direct handler, tool registry, permission evaluator,
confirmation token, arbitrary shell, Git publication, or release authority.

## Compatibility and migration

Ordinary chat, contextual routing, manual Patch/Test commands, release tooling,
public trace APIs, v2.10/v2.11 trace data, and v2.12 diagnostics and repair remain
compatible. Old trace records omit the optional workflow decision field and
continue to round-trip.

Safely compatible v1 workflow state migrates to schema v2. `waiting_patch` can
resume after raw task/context removal. Ambiguous old executing or confirmation
state becomes `failed/state_incompatible`; it is never executed. Recovery is
state-only and requires a later explicit `resume`.

## Validation

The focused v2.13 gate, complete suite, compileall, identity, production policy,
smoke, built-in Release Manager, working/staged whitespace, and generated-state
hygiene are required and recorded in the pull request and published release.
GitHub Actions runs the full suite on Windows and Linux with Python 3.12, 3.13,
and 3.14 plus a focused Ubuntu/Python 3.12 release gate.

Local Windows/Python 3.12 validation recorded `233 passed, 4 skipped, 21
subtests` for the focused release set and `1017 passed, 7 skipped, 136 subtests`
for the complete suite. The seven skips are understood symlink-privilege
conditions: five pre-existing plugin, diagnostics, state-integrity, and terminal
cases plus two v2.13 workflow/file path cases. Compileall, identity, production
policy, smoke, Release Manager, whitespace, and generated-state hygiene passed.
The two production-policy warnings (`bug_fix` and `test_run` inactive automatic
routes) are intentional confirmation-only boundaries, not degraded issues.

## Known limitations

* Investigation and deterministic review rules are bounded, not proof of
  correctness.
* Patch content must be prepared through existing managed Patch Tools.
* Rollback deliberately refuses after unrelated workspace changes.
* Locks coordinate cooperating VEGA processes and remain advisory.
* Windows symlink cases skip explicitly when symlink privileges are unavailable.
* No background autonomy, arbitrary shell, automatic internet, telemetry,
  permission changes, Git writes, plugin installation, or in-product release
  publication was added.
