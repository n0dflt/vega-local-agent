# VEGA v2.10.0 Release Notes

VEGA v2.10.0 is a stabilization release focused on production reliability,
cross-layer consistency, deterministic failure behavior, and privacy-preserving
local diagnostics. It does not add autonomous tool execution or a new tool
execution path.

## Highlights

* One immutable validated production runtime snapshot for domains, intents,
  capabilities, built-in/plugin tools, permissions, routing, model profiles,
  and context budgets.
* Fail-closed startup gating: fatal configuration or executor-construction
  failures publish only a blocked executor and a safe diagnostic report.
* Strict `schema_version: 1` validation for routing and model policies alongside
  the existing versioned policy formats.
* Deterministic model/profile selection, installed-model fallback, bounded
  context selection, and evidence-preserving synthesis fallback.
* Bounded redacted execution traces and safe `/doctor trace latest` summaries.
* Git-safe runtime-state ownership with atomic model-profile replacement.
* Stable failure codes and safe user messages across analysis, planning,
  permissions, execution, synthesis, tracing, diagnostics, and snapshot loading.

## Execution trace contract

Tracing observes the existing path only:

```text
contextual runtime -> execute_plan() -> PlanExecutor -> ToolExecutor
```

The observer never calls a tool, handler, execution callback, or retry. Traces
contain only the documented machine-decision allowlist: trace/request type,
intent/domain, required capabilities, selected tools, permission outcomes,
confirmation flag, model decision, context-budget counters, bounded step
metadata, terminal status, and stable error codes.

Every identifier is limited to 128 characters. Steps and related collections
are limited to 8 items. Serialized records are limited to 256 KiB. Finalization
permits only `STARTED -> COMPLETED`, `STARTED -> BLOCKED`, or
`STARTED -> FAILED`; the recorder is cleared and rejects all later mutation.

Traces never include prompts, raw requests, evidence, file contents, tool
arguments or results, payloads, handlers, callbacks, confirmation tokens,
permission grants, environment variables, secrets, API keys, full local paths,
URL query parameters, exception objects/text, tracebacks, or arbitrary object
fields. Serialization is manual, allowlisted, and fail closed.

## Opt-in local diagnostics

Persistence is disabled by default. It is enabled only when
`VEGA_EXECUTION_TRACE` is `1`, `true`, `yes`, or `on` (case-insensitive).
Enabled traces append as UTF-8 JSONL to:

```text
logs/diagnostics/execution-traces.jsonl
```

The active file rotates before exceeding 5 MiB and retains one backup:

```text
logs/diagnostics/execution-traces.jsonl.1
```

Both paths are ignored by Git. Serialization, callback, directory creation,
append, and rotation failures are diagnostic-only and cannot change a user
response or plan result. Locking is process-local; v2.10 does not claim
interprocess-safe appends.

## Failure-path and cleanup behavior

* Raw exception text is replaced by stable error codes and safe messages.
* Permission denial or missing confirmation prevents handler invocation.
* Whole-plan preflight prevents partial execution when a later step is unsafe.
* Tool execution stops after the first failure and is never retried by tracing.
* Failed synthesis keeps a completed plan `COMPLETED`, adds only
  `synthesis_failed`, and returns the deterministic evidence response.
* A fatal production snapshot creates a safe `BLOCKED` diagnostic trace.
* Request-local recorder state never stores arguments, payloads, results, or
  handlers and is cleared after terminal finalization.
* `data/model_profile.json` and `data/project_state/tasks.json` are preserved as
  local ignored state rather than repository content.

## Migration impact

* Routing and model policy fixtures must include integer `schema_version: 1` and
  must not contain unknown fields.
* Explicit model overrides synthesize only when the model appears in the
  supplied installed-model set; otherwise deterministic tool output is used.
* Tool/signature/argument/permission exceptions now expose stable safe messages
  instead of raw exception details.
* Existing local model-profile and task files are preserved. They are no longer
  tracked and normal profile switching should not dirty the Git working tree.

## Known limitations

* The Plugin API loads explicitly trusted Python and is not a sandbox; import and
  plugin-factory side effects cannot be rolled back completely.
* `bug_fix` and `test_run` remain confirmation-only and outside the automatic
  contextual catalog. Policy consistency reports them as nonblocking warnings.
* Trace retention is fixed at one backup and is not configurable in v2.10.
* Trace appends use process-local locking only; concurrent processes are not
  coordinated.
* Payload-free traces explain routing and policy decisions, not content quality.
* There is no trace export, remote telemetry, GUI diagnostics, marketplace, or
  automatic publishing.

## Verification

The release is validated with focused execution-trace, contextual runtime,
planning, production snapshot, model, permission, doctor, persistence,
rotation, cleanup, cross-layer, failure-path, and release-identity tests,
followed by the complete pytest suite, `compileall`, and `git diff --check`.
