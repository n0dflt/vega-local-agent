# VEGA v2.12 — Local State Integrity & Recovery

Maintained release: `v2.12.1` (architecture introduced in `v2.12.0`)

## Scope

v2.12 hardens only VEGA-generated execution traces and diagnostic reports. It
adds bounded coordination between local VEGA processes and an explicit recovery
workflow for interrupted or corrupted generated state. It does not add a tool,
model call, background task, upload, telemetry path, or autonomous repair.

The supported execution route remains:

```text
contextual runtime
    -> plan execution
    -> PlanExecutor
    -> ToolExecutor
```

Diagnostics observe this route and never receive `ToolExecutor` or tool
authority.

## Generated-state boundaries

The strict diagnostics policy schema version 2 validates these project-relative
locations before use:

* `trace_store_path` for the active JSONL file and numbered backups;
* `doctor_reports_dir` for atomic report exports;
* a fixed `quarantine` child of the validated trace directory;
* fixed `.trace-state.lock` and `.report-state.lock` names inside their generated
  directories.

Absolute, drive-qualified, parent-traversing, root-escaping, symlink-escaping,
`.git`, virtual-environment, cache, and `node_modules` paths are rejected. Repair
does not accept a path, recurse, or act on filenames outside its compiled
allowlists. Generated lock, temp, trace, report, and quarantine files are ignored
by Git.

## Cross-process coordination

`GeneratedStateLock` retains bounded process-local locks and adds an operating
system file lock:

* Windows uses a non-blocking one-byte `msvcrt.locking` range;
* POSIX uses non-blocking advisory `fcntl.flock`;
* acquisition retries only until `lock_timeout_ms`, capped at 5,000 ms;
* fixed lock files remain in place, avoiding delete/recreate inode races;
* context-manager cleanup releases locks after success, exceptions, and failure
  injection; an exited process releases its OS-owned lock;
* acquisition, timeout, use, and release failures map to fixed codes.

Trace append, threshold rotation, report replacement, and report retention run
under the appropriate lock. Trace data and reports are flushed and `fsync`ed.
Lock failures are best-effort persistence failures and cannot change plan, tool,
or user-visible execution results.

Locks are advisory. A non-VEGA process that ignores the protocol can still alter
generated files.

## Inspection model

`/doctor state status` performs a read-only, bounded inspection. It does not
create a directory or lock file. If a fixed lock file already exists, status
probes it without waiting or exposing owner details. The immutable slotted
`StateIntegrityDiagnostics` model manually serializes only:

* lock availability;
* stale recognized temp-file count;
* recoverable torn-tail boolean;
* corrupted generated-file count;
* quarantined-file count;
* scan-limit boolean;
* normalized relative paths;
* fixed safe codes.

No prompts, user content, trace payloads, exception text, tracebacks, absolute
paths, environment values, tokens, or arbitrary representations cross the
diagnostics boundary.

## Recovery classification

Inspection and repair distinguish these states:

1. A final JSONL fragment without `\n` is a recoverable torn tail. Valid earlier
   records are preserved. A valid final record missing only the newline is
   normalized; an invalid incomplete fragment is removed.
2. An invalid newline-terminated record is complete, ambiguous corruption. The
   original generated file is quarantined before a repaired file containing its
   valid records atomically replaces it.
3. An oversized generated trace is atomically moved to quarantine without
   reading its payload.
4. A malformed generated diagnostic report is quarantined, then removed from the
   report directory. Unrecognized files are preserved.
5. Only exact same-directory atomic-write temp names older than
   `stale_temp_age_seconds` are stale and eligible for deletion.

All writes use same-directory exclusive temp files, flush, `fsync`, and atomic
replacement. A quarantine name includes only a sanitized generated filename and
a bounded digest or random identifier. Existing content-addressed quarantine
copies make repeated repair idempotent. Retention removes only exact quarantine
names.

## Explicit command contract

The only state commands are exact, case-sensitive strings:

```text
/doctor state status
/doctor state repair
```

Status is read-only. Repair accepts no arguments, acquires locks in the fixed
trace-then-report order, re-inspects under those locks, and mutates only
recognized generated state. Unknown, fuzzy, differently cased, extra-whitespace,
absolute-path, relative-path, and additional-argument variants print deterministic
doctor usage and do not mutate the filesystem.

Repair never runs during startup. It is bounded and safe to repeat.

## Resource limits and policy

Compiled caps bound lock timeout, trace/report bytes, trace line length, trace
records, trace backups, files scanned, temp cleanup, retained reports, retained
quarantine files, and directory entries visited. Configured values are complete,
strictly typed, positive where required, and cannot exceed these caps. Boolean
values are not accepted as integers. Unknown, missing, and duplicate fields fail
closed.

The fixed v2.12 state vocabulary is:

```text
state_lock_unavailable
state_lock_timeout
state_lock_operation_failed
stale_temporary_state
recovered_torn_trace_tail
corrupt_generated_state
quarantine_failed
state_repair_failed
state_scan_limit_reached
```

## Compatibility and migration

The public trace functions and v2.10/v2.11 allowlisted trace record shape are
unchanged. Existing valid active and backup files remain readable. Trace
persistence remains opt-in through `VEGA_EXECUTION_TRACE`. Upgrading requires no
state migration: policy schema version 2 adds lock timeout, stale-temp age,
state-scan, and quarantine-retention values, while repair remains user-invoked.

Downgrading application code requires restoring the v2.11 diagnostics policy
schema. Valid traces remain compatible. Quarantine and fixed lock files may be
left in place or removed while VEGA is stopped; they are ignored by Git. Do not
restore a quarantined file over a valid trace without manual inspection.

## Threat model and guarantees

The design protects against concurrent cooperating VEGA processes, interrupted
writes, bounded generated-file corruption, hostile filenames in generated
directories, policy path escape, symlink substitution detected before use, and
payload leakage through diagnostics. It fails closed on unsupported state,
limits, lock failures, quarantine failures, and repair failures.

It is not a filesystem sandbox, malware scanner, distributed lock service, or
forensic repair tool. It cannot guarantee integrity against a privileged local
attacker, storage hardware failure, or unrelated processes that ignore advisory
locks. Windows lock-file symlink rejection uses the platform checks available to
the Python standard library; POSIX additionally uses `O_NOFOLLOW` when opening a
lock file.

## Validation

The release gate covers multiprocessing append and rotation, contention and
timeouts, abandoned locks, adapter behavior, exception cleanup, stale temp
handling, torn tails, complete corruption, quarantine confinement/retention,
oversized state, scan caps, symlink/path rejection, exact command routing,
secret sentinels, v2.10/v2.11 compatibility, report atomicity, policy
consistency, release identity, compileall, smoke tests, Release Manager, Git
whitespace checks, generated-state hygiene, and the full test suite.
The final Windows gate recorded `106 passed, 2 skipped` focused results and
`1126` full-suite test results with zero failures, zero errors, and five
symlink-privilege skips. Compileall, identity, policy consistency, smoke, Release
Manager, Git whitespace, and generated-state tracking checks passed. Four of the
five conditional skips predate v2.12; the new state-integrity symlink regression
is skipped under the same unavailable Windows privilege.
